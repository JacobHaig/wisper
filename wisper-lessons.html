Lesson 00

The map: one model, three orchestrations

Win: after this you can name every file and say why it exists in one sentence.

Here is the single most important idea in the whole repository. Every one of the three modes ultimately runs the same line of code:

asr_model = asr.models.ASRModel.from_pretrained("nvidia/parakeet-tdt-0.6b-v3")
outputs   = asr_model.transcribe([path], timestamps=True)

This exact call appears in main.py:342/351, realtime_asr.py:100/228, and main_single_track.py:129/135.

Why this matters. If the model call is identical everywhere, then all the interesting engineering in this repo is not about ASR at all. It is about (a) getting audio to that call and (b) assembling what comes back. Keep that lens on as you read.

The model is a black box that turns a chunk of audio into timestamped words. Everything else wisper contains exists to feed that box or to organize its output. That reframes ten files into two jobs.

RecallAcross all three modes, what actually differs between them?

Right: the plumbing, not the model. All three call the identical Parakeet TDT model with timestamps=True. What varies is the input path (whole file, live buffer, streaming buffer) and the output assembly (merge, dedup, speaker-tag). Hold this idea — it makes every other lesson shorter.

Playbook · how to actually run it

Running wisper

Win: you can go from a fresh checkout to a transcript — and to a live feed — with copy-paste commands.

Setup (once)

uv sync --extra cu128

Installs all dependencies plus CUDA 12.8 PyTorch from the custom index (pyproject.toml:19–35). Every command below uses uv run, which runs inside that environment.

Prerequisites the code assumes: Python 3.12, an NVIDIA CUDA 12.8 GPU, and ffmpeg + ffprobe on your PATH — the batch and network paths shell out to them directly (main.py:156, audio_capture.py:203).

Windows gotcha (the real one) Use Python 3.12, not 3.13. On 3.13 there's no prebuilt editdistance wheel (a NeMo dependency), so uv compiles it from source with Cython — and Windows 11 Smart App Control blocks the unsigned build DLL with ImportError: DLL load failed … An Application Control policy has blocked this file. On 3.12 the wheel is prebuilt, so nothing compiles and nothing gets blocked. If a stale environment picks the wrong version: uv python pin 3.12, delete the .venv folder, then re-run uv sync --extra cu128.
First run is slow. The Parakeet model (~0.6B params) downloads from HuggingFace the first time it's loaded, then caches. A first transcription can take minutes before any output appears; later runs skip the download.

Mode A — Batch transcription (main.py)

The everyday path: drop media in video/, run, collect JSON from transcript/.

uv run main.py                                 # all videos in video/
uv run main.py -i video/interview.mp4          # one specific file
uv run main.py --audio-only -i audio/pod.mp3   # skip video extraction
uv run main.py --format txt                    # plain text (segments + timestamps)
uv run main.py --format plain                  # plain text, NO timestamps (segment text only)
uv run main.py --youtube "https://youtu.be/XXXX"   # download from YouTube, then transcribe
uv run main.py --tracks 1 2                     # only these audio tracks (1-indexed)
uv run main.py --skip-existing --dry-run        # preview what would run, do nothing
uv run main.py --chunk-minutes 0                # disable chunking (short files / big GPU)
uv run main.py -v                               # verbose: print segments as they're recognized

Mode B — Real-time service (serve.py)

Starts audio capture + ASR workers + a web server that stays running. Terminal 1 — launch it:

uv run serve.py                                    # microphone, port 8000
uv run serve.py --source desktop                   # desktop audio (WASAPI loopback, Windows)
uv run serve.py --source both --mix mixed          # mic + desktop combined into one stream
uv run serve.py --source both                      # mic + desktop as two separate feeds
uv run serve.py --stream-url rtp://0.0.0.0:5004    # network stream (auto-sets --source=network)
uv run serve.py --port 9000 --chunk-seconds 5      # custom port, 5s inference window

Options in build_parser, serve.py:34–86. Leave this running; query it from elsewhere.

Terminal 2 (or your browser) — read the live transcript. GET endpoints work by pasting the URL straight into a browser:

http://127.0.0.1:8000/api/status          # is it running? which sources? word counts
http://127.0.0.1:8000/api/last-word       # most recent word
http://127.0.0.1:8000/api/last-words?n=50 # last 50 words (rolling — re-readable)
http://127.0.0.1:8000/api/last-segment    # most recent full sentence/segment
http://127.0.0.1:8000/api/pop-words?n=10  # next 10 words, CONSUMED on read
peek vs. consume. last-* peeks the rolling window (read it as often as you like — good for a dashboard). pop-* drains a mailbox (each word returned once — good for a downstream process). Same idea as Lesson 05.
Separate mode. With --source both (not mixed) there are two feeds; add ?source=mic or ?source=desktop to any endpoint, e.g. /api/last-word?source=desktop.

For a continuous push feed instead of polling, open the WebSocket. Paste this in a browser console:

const ws = new WebSocket("ws://127.0.0.1:8000/ws/stream");
ws.onmessage = e => console.log(JSON.parse(e.data));   // {timestamp, word, source}
Two gotchas. (1) The server binds 127.0.0.1 (localhost only) by default — pass --host 0.0.0.0 to reach it from another machine on your network. (2) On Windows PowerShell, curl is an alias for a different tool; use curl.exe, Invoke-RestMethod, or just the browser for the GET endpoints above.

uv run main_single_track.py                        # iterates video/, writes transcript/

Plays a saved JSON transcript back word-by-word at its real timing (via time.sleep) — a quick eyeball check that timestamps look sane. Edit the hardcoded path first:

# test_transcript.py:34  →  set input_path to your file

ApplyYou want a plain-text transcript of one long podcast MP3, and your GPU is modest. Which command?

Right. --audio-only skips video extraction for an MP3, --format txt gives plain text, and leaving --chunk-minutes at its 5-minute default keeps a modest GPU from running out of memory on a long file. Option 2 disables the very chunking that protects a small GPU; serve.py is the live service (no --format); and main_single_track.py has no such flags.

Lesson 01

The shared vocabulary

Win: you can explain why models.py is the quiet keystone of the whole project.

NeMo hands back untyped dictionaries: output.timestamp["word"] is a list of {"start", "end", "word"} dicts. Loose dicts are fine for one script, but this repo has three pipelines that all need to agree on shape. So the first thing every pipeline does is translate NeMo's dicts into typed dataclasses.

@dataclass(frozen=True, slots=True)
class WordTimestamp:
    start: float
    end: float
    word: str

models.py:6–10. Also SegmentTimestamp, CharTimestamp, the container TrackTranscript, and the real-time LiveWord.

Two words to learn. frozen=True makes instances immutable (can't accidentally rewrite a timestamp after the fact). slots=True drops the per-instance __dict__, saving memory — it matters when a long file produces tens of thousands of word objects. See dataclasses docs.

This is the seam of the codebase: the one type that batch and real-time both route through. The conversion happens in main.py:270–277 (_output_to_transcript) for batch, and produces LiveWord objects in realtime_asr.py:244 for live mode.

Notice the two container shapes serve different lifecycles:

  • TrackTranscript — a whole finished transcript (three parallel lists: word, segment, char). Batch mode's unit. It even carries its own to_dict() for JSON serialization (models.py:33).
  • LiveWord — a single word, the instant it's recognized, tagged with timestamp and a source label ("mic"/"desktop"/"mixed"). Real-time mode's unit, because live mode ships one word at a time.

UnderstandWhy does WordTimestamp use frozen=True, slots=True?

Right. frozen = safety (a recognized word shouldn't mutate); slots = memory (thousands of tiny objects per long transcript). Neither is about speed of inference or serialization.

Lesson 02

The batch pipeline & the chunking trick

Win: you can explain how a 2-hour file gets transcribed without blowing up the GPU.

The batch pipeline (main.py) is a clean four-step assembly line, visible in video_to_transcript (main.py:406):

  1. Detect & extract tracks — ffprobe counts audio streams, then ffmpeg exports each as a mono MP3 (convert_video_to_audio_tracks, main.py:285). A video can carry several audio tracks; each becomes its own transcript.
  2. Transcribe — load Parakeet, run it (transcribe_audio_parakeet, main.py:331).
  3. Convert — NeMo dicts → TrackTranscript.
  4. Save — JSON (save_transcript_to_file) or timestamped text (save_transcript_as_text).

The real problem: GPU memory

ASR models hold the whole audio's activations in VRAM. Feed a 90-minute file in one pass and CUDA runs out of memory. The fix is chunking: cut the audio into N-minute pieces, transcribe each, stitch the results back. But a naïve cut slices words in half at the boundaries. So the chunks overlap.

stride = chunk_seconds - overlap_seconds     # main.py:188
# ... windows: [0, chunk], [stride, stride+chunk], ...
ffmpeg -ss {start} -t {chunk_dur} -i in.mp3 -c copy chunk.mp3

split_audio_into_chunks, main.py:172–214. Overlap is a fixed 15 s (CHUNK_OVERLAP_SECONDS).

-c copy. ffmpeg copies the compressed MP3 stream instead of re-encoding it — near-instant, no quality loss. The tradeoff: cuts land on codec frame boundaries, not exact samples. For chunking that's harmless; for sample-accurate editing it wouldn't be.

Stitching without duplicates

Overlap means the last 15 s of chunk i and the first 15 s of chunk i+1 both get transcribed. If you naïvely concatenated, you'd repeat those words. The merge (merge_chunk_transcripts, main.py:217) draws a cut line at the midpoint of each overlap window and keeps each word from exactly one side:

cuts[i] = next_offset + overlap_seconds / 2      # main.py:234
# a word survives if:  lower <= (word.start + offset) < upper
Why the midpoint? A word near a chunk's edge was transcribed with little surrounding context, so it's more error-prone. The midpoint heuristic keeps each word from the chunk in whose interior it fell — where the model saw the most context around it.

Every timestamp is also shifted by that chunk's offset so the merged transcript reads as one continuous timeline. The output is structurally identical to a non-chunked run — that's the whole point. Set --chunk-minutes 0 to disable it (main.py:439).

UnderstandWhy do the audio chunks overlap by 15 seconds?

Right. A hard cut could split a word across chunks and mangle both halves. Overlap guarantees every word appears whole in some chunk; the midpoint cut line then decides which copy to keep so it isn't duplicated.

Lesson 03

Real-time: producer, consumer, and backpressure

Win: you can trace a single spoken word from microphone to recognized text.

Live transcription is a classic producer/consumer problem. Audio arrives continuously and can't be paused; inference is slower and bursty. The buffer between them is a queue.Queue.

self.mic_queue: queue.Queue[np.ndarray] = queue.Queue(maxsize=60)

audio_capture.py:115. At the default 0.5 s chunk, 60 slots ≈ 30 s of buffered audio.

Producer (capture) pushes ~0.5 s audio chunks. Consumer (the ASR worker) drains them. When they fall out of sync, the bounded queue is the pressure valve — and this repo makes a deliberate choice about what to do when it fills:

try:
    target_queue.put_nowait(audio)
except queue.Full:
    target_queue.get_nowait()      # drop the OLDEST chunk
    target_queue.put_nowait(audio) # keep the newest

audio_capture.py:156–167 (same pattern for the network reader at :248).

Freshness > completeness. A live transcriber that falls behind should skip stale audio and stay current, not queue up a growing backlog it will narrate minutes late. Dropping the oldest chunk is the correct choice for this use case — and the opposite of what a batch job (which must transcribe everything) would want.

The consumer's loop

The worker (ChunkedParakeetWorker._run, realtime_asr.py:163) does a small state machine on its own thread:

  1. Drain everything currently in the queue into a running buffer (:176).
  2. Wait until the buffer holds at least chunk_seconds (default 3 s) of audio (:186).
  3. Transcribe the first 3 s; then keep the last 1 s as overlap for the next round: buffer = buffer[chunk_samples - overlap_samples:] (:190–192).
  4. Dedup & emit — push only genuinely new words into the TranscriptStore (:197). How dedup works is Lesson 04.
Two chunk sizes, don't confuse them. Capture chunk = 0.5 s (how often audio is delivered). Worker chunk = 3 s (how much audio each inference sees). The worker accumulates ~6 capture chunks before it runs the model once.

There's also a graceful-shutdown tail: on stop, any leftover buffer ≥ 0.5 s gets one final transcription (realtime_asr.py:202) so the last words aren't lost.

DesignWhen the audio queue is full, the code drops the oldest chunk. Why is that right here?

Right. This is a latency-vs-completeness tradeoff. A real-time transcriber that buffers a backlog drifts further behind reality every second. Better to skip stale audio and report what's happening now. A batch pipeline would make the opposite choice.

Lesson 04 · the keystone lesson

One problem, two solutions: overlap & dedup

Win: you can contrast how batch and real-time solve the same duplication problem — and say why each fits its context.

Both pipelines overlap their chunks (Lessons 02 & 03) for the same reason: don't slice words at boundaries. Both therefore transcribe the overlap region twice and must remove the duplicates. But they have different information available, so they solve it differently. This contrast is the heart of the repo.

Batch: cut by time

Batch knows every word's absolute timestamp, so it can reason spatially: draw a line at the overlap midpoint and keep whichever words fall on the correct side (Lesson 02). Clean, deterministic, timestamp-driven.

Real-time: cut by text

The live worker can't compare against "the next chunk" — that audio doesn't exist yet. All it has is a memory of the words it already emitted. So it dedups by matching word sequences: find the longest run where the tail of what-was-said equals the head of what-was-just-heard, and skip it.

@staticmethod
def _find_overlap(emitted, new):
    max_check = min(len(emitted), len(new), 20)
    for length in range(max_check, 0, -1):     # longest match first
        if emitted[-length:] == new[:length]:
            return length
    return 0

realtime_asr.py:268–285. The returned count is skipped in _transcribe_chunk (:243).

Greedy longest-first. It tries the longest possible overlap (up to 20 words) and returns the first match, so a short accidental coincidence ("the… the") can't beat a genuine long overlap. emitted_words is also trimmed to ~100 entries (:253) so the match window can't grow without bound.

One more subtlety worth spotting: segments get a cruder skip, raw_segments[skip and 1 or 0:] (realtime_asr.py:257) — a terse pre-ternary Python idiom meaning "if we skipped any words, drop the first segment too (it's probably the overlap), else keep all." Words get precise dedup; segments get a heuristic.

The insight to keep. "Overlap to avoid cutting, then dedup the overlap" is one pattern with two implementations. Batch has global timestamps → cut by position. Streaming has only history → cut by matching text. Same problem, different information available, different solution. Recognizing this shape will help you far beyond this repo.

ContrastWhy can't real-time mode use the batch merge (midpoint-of-overlap cut)?

Right. The batch cut compares two adjacent, already-transcribed chunks. Live audio arrives incrementally — there is no "next chunk" to draw a midpoint against. So the worker dedups against the only thing it has: the words it already emitted, matched as text.

Lesson 05

Wiring it together: three concurrency domains, two seams

Win: you can draw who runs on which thread and where they hand off data.

The real-time service has three things running at once, in different concurrency models:

  • Capture callbacks — run on PortAudio's own thread(s) (or an ffmpeg-reader thread for network). Producers.
  • ASR worker(s) — a plain threading.Thread per source (realtime_asr.py:81). Consumers.
  • uvicorn / FastAPI — an async event loop serving HTTP + WebSocket.

These never call each other directly. They meet at exactly two objects: the queue.Queue (capture → worker) and the TranscriptStore (worker → web).

The store is three data structures in a trench coat

TranscriptStore (transcript_store.py) serves three different read patterns, so it keeps three structures — and add_word writes to all of them (:32):

StructurePurposeRead by
deque(maxlen=10000)Rolling "last N words" window/api/last-word, /api/last-words
queue.Queue (unbounded)Consume-once mailbox/api/pop-word, /api/pop-words
list[asyncio.Queue]One live feed per WS clientWS /ws/stream

transcript_store.py:20–27. The deque and segment access are guarded by _lock; the subscriber list by _ws_lock.

pop vs. last. pop-* endpoints consume words (each returned once — good for a downstream consumer draining the feed). last-* endpoints peek at the rolling window (same word can be read repeatedly — good for a dashboard). Two structures, two semantics, deliberately.

Dependency injection by module attribute

server.py defines the app and endpoints but starts with empty globals: stores = {} (:23). serve.py fills them in at startup through a FastAPI lifespan context manager, then starts capture and workers, and reverses it all on shutdown:

@asynccontextmanager
async def lifespan(app):
    import server
    server.stores = stores          # inject the wired-up stores
    server._status_info = {...}
    capture.start()
    for worker, q in workers: worker.start(q)
    yield                            # ← app serves requests here
    for worker, _ in workers: worker.stop()
    capture.stop()

serve.py:153–188. This is why server.py can be imported without any hardware attached — the wiring is deferred to run time.

Sharp edge — verify add_word calls put_nowait on an asyncio.Queue from the worker thread (transcript_store.py:38). asyncio.Queue is not documented as thread-safe; the textbook-correct bridge is loop.call_soon_threadsafe(...). It works in practice here, but if you ever add features to the WebSocket path, this is the line to scrutinize. Good thing to confirm against the asyncio docs.

UnderstandWhat is the role of the lifespan context manager in serve.py?

Right. The lifespan runs setup before the app serves (inject stores, start capture + workers) and teardown after (stop everything). It's the seam between "an importable FastAPI app" and "a running service with live hardware."

Lesson 06

Capturing the physical world

Win: you can explain how three very different sources become one uniform stream.

The worker (Lesson 03) doesn't care where audio came from — it just wants float32, 16 kHz, mono chunks. AudioCapture's whole job is to make mic, desktop, and network all look identical. That's a textbook abstraction boundary: hide three messy realities behind one clean contract.

SourceMechanismWhere
Micsounddevice.InputStream callback_open_mic_stream :264
DesktopWASAPI loopback (Windows-only)_open_desktop_stream :281
Networkffmpeg subprocess piping raw PCM_run_network_reader :220

The two conversions every source must do

1. Sample-rate match. A mic might run at 44.1 or 48 kHz natively, but NeMo wants 16 kHz. The callback resamples with a rational ratio computed via GCD:

g = gcd(self.sample_rate, native_rate)
audio = resample_poly(audio, self.sample_rate // g, native_rate // g)

audio_capture.py:151–154. e.g. 48000→16000 reduces to up=1, down=3.

2. Format match. The network path receives 16-bit signed PCM bytes from ffmpeg and must convert to normalized float:

audio = np.frombuffer(raw, dtype=np.int16).astype(np.float32) / 32768.0

audio_capture.py:247. Dividing by 215 maps int16's ±32768 range onto float [-1, 1).

WASAPI loopback records what your speakers are playing — the OS routes the output back as a capture device. _find_wasapi_loopback_device (:30) hunts for it; if it can't, it raises a helpful error telling you to pass --device-index. This is Windows-only, hence the explicit sys.platform != "win32" guard at :282.
Two capture strategies. Mic/desktop use push (sounddevice invokes your callback). Network uses pull (a background thread blocks on proc.stdout.read()). Different mechanisms, same destination queue — the abstraction holds.

The get_queues() method (:124) is the routing table: it decides, based on source and mix_mode, which named queues exist. --source both --mix separate yields two queues (mic, desktop) → two workers → two independent transcripts; --mix mixed yields one shared queue both devices write into.

UnderstandWhat does the abstraction in AudioCapture guarantee to the ASR worker?

Right. The point of the abstraction is that the worker is source-agnostic. Mic, WASAPI loopback, and ffmpeg network streams all resample and reformat down to the same float32/16 kHz/mono contract before hitting the queue.

Lesson 07

Diarization mode (who spoke when)

Win: you can say what this mode adds over the others, and why its config is so big.

main_single_track.py is the exploratory pipeline for multi-speaker conversations. Plain transcription tells you what was said; diarization adds who said it. It composes two additional NVIDIA models on top of Parakeet:

  • nvidia/diar_streaming_sortformer_4spk-v2.1 — Sortformer, a streaming diarizer that segments audio by speaker (up to 4).
  • nvidia/multitalker-parakeet-streaming-0.6b-v1 — a multitalker ASR variant that transcribes with speaker awareness.

The core loop (transcribe_audio_multitalker_parakeet, :61) feeds audio through a CacheAwareStreamingAudioBuffer and a SpeakerTaggedASR helper, chunk by chunk, producing speaker-tagged segments. That "cache-aware streaming" is how the model processes a long stream in pieces while remembering context across chunk boundaries — the streaming analogue to Lesson 02's overlap trick, but handled inside NeMo.

Why ~50 config knobs? MultitalkerTranscriptionConfig exposes streaming diarization internals: spkcache_len, fifo_len, chunk_left/right_context, and more. Streaming diarization has to decide how much past audio to remember per speaker and how much look-ahead it's allowed — each is a latency/accuracy dial. Contrast this with batch wisper, which has one real knob (--chunk-minutes).
Honest framing. This file carries an NVIDIA license header and mirrors NVIDIA's reference multitalker example — it's the least "yours" and most experimental code in the repo (note the leftover stray imports at the top, e.g. unittest.result). Treat it as an integration of NVIDIA's streaming diarization recipe, not hand-built infrastructure like the real-time service. If you extend it, the Sortformer model card is your primary reference. Verify against upstream.

RecallWhat does diarization add that plain transcription does not?

Right. Diarization answers "who spoke when." Sortformer segments the audio by speaker; the multitalker ASR then attributes transcribed text to those speakers.

Reference

Glossary

The shared vocabulary — adhered to across every lesson above.

ASR
Automatic Speech Recognition — turning audio into text. The core task of this repo.
Parakeet TDT
NVIDIA's ASR model family used here (parakeet-tdt-0.6b-v3). TDT = Token-and-Duration Transducer, a transducer variant that predicts both tokens and how long each lasts, which is what gives the clean word/segment/char timestamps.
NeMo
NVIDIA's toolkit that hosts the models. This repo uses it only through ASRModel.from_pretrained(...).transcribe(...).
Diarization
Determining who spoke when — speaker segmentation. Provided by Sortformer in Lesson 07.
Chunking
Splitting long audio into pieces to fit GPU memory (batch) or to enable incremental processing (streaming).
Overlap
Deliberately sharing audio between adjacent chunks so words at boundaries aren't sliced. Requires later dedup.
Producer / consumer
A pattern where one party generates data (audio capture) and another processes it (ASR worker), decoupled by a queue.
Backpressure
What a bounded queue exerts when the consumer can't keep up. Here, resolved by dropping the oldest chunk to favor freshness.
WASAPI loopback
A Windows audio feature that lets you record what the speakers are playing, by exposing the output as a capture device.
Resampling
Converting audio from one sample rate to another (e.g. 48 kHz mic → 16 kHz for NeMo) via scipy.signal.resample_poly.
Lifespan
A FastAPI startup/shutdown context manager. Here it injects shared state and starts/stops the capture and workers.
Dependency injection
Supplying an object its collaborators from outside rather than having it construct them. serve.py injects stores into server.py.
Seam
A shared type or boundary many parts route through. Here: TrackTranscript/LiveWord and the TranscriptStore.
